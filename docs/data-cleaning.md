# Data Cleaning with Claude Code — Process Overview

This document explains, end to end, how raw public-data files become cleaned,
contract-described, served datasets in this project — and how you could
implement the same process yourself with Claude Code. The through-line is
**Claude skills**: each stage of the pipeline is encoded as a skill (a
versioned, checked-in prompt with a fixed procedure), so the work is
repeatable, reviewable, and hard to get subtly wrong.

If you only read one thing, read this: **the human never edits the cleaned
data by hand.** Every change flows through a Python `transform.py` that a skill
authors and re-runs, and the transform *validates itself* before it is allowed
to succeed. The skills are the guardrails that make that discipline hold across
dozens of datasets.

---

## 1. The mental model

### Bronze → Gold (the medallion pattern)

Data moves through two tiers:

| Tier | What it is | Where it lives | Editable? |
|------|-----------|----------------|-----------|
| **Bronze** | Raw source files exactly as published (CSV/XLS/XLSX), byte-for-byte | `data/bronze/{main_topic}/{sub_topic}/{topic}/` | **Never touched** after download |
| **Gold** | Cleaned, typed, standardized Parquet in a star schema | `data/gold/{main_topic}/{topic}/` | **Never edited directly** — only regenerated by `transform.py` |

Bronze is the immutable source of truth about *what was published*. Gold is the
cleaned product. The **cardinal rule** is that gold is a pure function of
bronze: you change gold only by changing the transform code and re-running it.
This means every value in gold is traceable back to a bronze cell, and a review
can always ask "does this gold value faithfully represent that bronze value?"

### The star schema (how gold is shaped)

Gold is not one big table. It is a **star schema**:

- **Fact tables** — lean tables of `year` + geography keys + `demographic` (if
  applicable) + topic categoricals + **metrics only**. No names, no labels, no
  descriptive text.
- **Dimension tables** — shared lookup tables that carry the human-readable
  labels (district names, school names, demographic labels, county names) and
  cross-dataset link keys. Fact tables join to them on a **natural key** (e.g. a
  district code), never a surrogate integer.

Keeping names out of fact tables is what lets many datasets share one set of
dimensions and link to each other on common geography.

### The data contract (how gold is described)

Every cleaned topic ships with a machine-readable **ODCS data contract**
(`contracts/{main_topic}/{topic}.odcs.yaml`) — the single schema artifact that
describes the columns, types, units, valid value ranges, the row grain, the
foreign-key joins, and the topic's "headline" metric. Critically, the contract
is **emitted by the transform itself**, not written separately. The same
contract is then consumed downstream by the API, the validator, and the
dashboard, so there is exactly one source of truth for a topic's schema.

## 2. Why skills (and what a skill is here)

A **Claude skill** is a checked-in Markdown file (`.claude/skills/<name>/SKILL.md`)
with YAML frontmatter and a fixed procedure. When you type `/bronze-data-structure`
in Claude Code, Claude loads that file and follows its steps. Think of a skill as
**a standard operating procedure that an agent executes** — with the exact bash
commands, the decision rules, the output format, and the "stop here if X" gates
all spelled out.

Why encode the data-cleaning process as skills rather than just prompting ad hoc?

1. **Repeatability.** The same dataset processed twice yields the same result,
   because the procedure is fixed and version-controlled, not improvised.
2. **Reviewability.** The procedure lives in git. When the process improves, you
   edit the skill once and every future run benefits.
3. **Enforced discipline.** Skills bake in the non-negotiable gates — freshness
   checks, self-validation, "never edit gold directly" — so an agent in a hurry
   can't skip them.
4. **Separation of concerns.** Authoring, reviewing, and approving are *different
   skills run by different agents* (often with fresh context), so the reviewer
   isn't just rubber-stamping its own work.
5. **Composability.** Small skills compose into orchestrators. `/full-pipeline`
   is just the single-topic skills wired together in order; `/batch-pipeline`
   fans `/full-pipeline` across many topics.

The rest of this document walks the pipeline stage by stage. Each stage names
the skill that runs it.

## 3. The skill catalog

| Stage | Skill | Role |
|-------|-------|------|
| 0 | *(a per-source `download.py`)* | Acquire raw files → bronze. Not a skill; plain Python. |
| 1 | `/bronze-data-structure` | Profile the raw files; produce `bronze-data-structure.md` (schema map + provenance + checksums). |
| 2 | `/transform-topic` | Author + run `transform.py` → gold Parquet + contract + manifest; **validates itself**. |
| 3a | `/data-review-claude` | Judgment-layer accuracy audit (bronze vs gold) → `data-review-claude.md`. |
| 3b | `data-review-codex` | Independent second audit by a different agent (Codex) → `data-review-codex.md`. |
| 4 | `/fix-from-reviews` | Apply the reviews' Required Fixes to `transform.py`, re-run, verify each fix landed. |
| 5 | `/approve-topic` | *(User only)* Flip `approved: true`; capture hash baselines. |
| — | `/full-pipeline` | Orchestrator: stages 1-gate → 2 → 3 → 4 for one topic. |
| — | `/batch-pipeline` | Run `/full-pipeline` across every eligible topic in parallel batches. |
| — | `/next-topic` | Pick the next unprocessed topic and run `/full-pipeline` on it. |
| — | `/pipeline-status` | Report which stages each topic has completed + drift status. |
| — | `/review-transform` | Lightweight code-quality review of a `transform.py` (no data checks). |
| — | `data-cleaning-standards` | Not a step — the **reference skill** of universal cleaning rules that stages 2-4 read. |

A companion reference skill, **`data-cleaning-standards`**, is not a step you
run; it is the rulebook (column naming, data types, percentage scaling,
demographics, suppression, the star schema, the required quality checks, the
canonical vocabulary). Stages 2, 3, and 4 all read it. When you adapt this
process to your own domain, this is the file you rewrite the most.

## 4. Stage 0 — Acquire the bronze data

**Not a skill — a plain Python `download.py` per source.**

Each data source has a small `download.py` (e.g.
`src/etl/criminal_justice/cdc/download.py`) whose only job is to fetch the raw
files exactly as the publisher serves them and drop them, unmodified, into the
bronze directory. Downloaders range from a trivial "GET these URLs" loop to a
scripted drive of a stateful HTML form (the CDC WONDER downloader replays a
multi-step form-POST session to extract county-grain mortality data the official
API refuses to return).

Two conventions matter here:

- **Filenames are normalized to snake_case** on the way in (`to_snake_case()` in
  `src/utils/bronze.py`), but the file *contents* are never altered.
- **Bronze is immutable.** From here on, nothing edits these files. If a source
  republishes, you re-download into bronze and re-run the downstream stages —
  the checksums (Stage 1) will detect the change and force a re-analysis.

The payoff of keeping bronze pristine: every later stage can trust that bronze
is "what the publisher said," and any discrepancy in gold is a *transform* bug,
never a "did someone tweak the source file?" mystery.

## 5. Stage 1 — `/bronze-data-structure`

**Skill: `bronze-data-structure`. Output: `bronze-data-structure.md` in the
topic's bronze directory.**

You cannot write a good transform for data you haven't profiled. This skill
does the profiling *once* and writes it down, so the transform-authoring stage
never has to re-explore the raw files from scratch.

Given a topic (`main_topic sub_topic topic`), the skill:

1. **Detects Excel sheet structure** — for each workbook, lists sheet names and
   classifies each sheet as Data / Metadata-Legend / Summary-Pivot, noting when
   the structure changes across years.
2. **Detects "eras"** — an *era* is a contiguous run of years whose files share
   the exact same column names. Datasets published over 15+ years almost always
   have several eras (columns get renamed, added, dropped), and era boundaries
   are where data-loss bugs cluster. The skill groups the files into eras and
   describes each era's columns.
3. **Profiles each era** — runs Polars `sample` / `describe` / `null_count` on a
   representative file per era.
4. **Classifies every string column** as truly *categorical* (list all distinct
   values) or *numeric-with-suppression* (list only the non-numeric suppression
   markers like `*`, `TFS`, `N/A`). This inventory is what the transform maps
   from.
5. **Runs the Asian / Pacific Islander check** — a specific, historically-costly
   trap: bronze that has only six race buckets and no separate Pacific Islander
   row may be using the pre-1997 combined "Asian" bucket. The skill applies a
   math test (do the race buckets sum to the total?) and flags the finding so
   the transform doesn't silently mislabel it.
6. **Identifies detail levels** (state / district / school) and **how the year
   is represented** (a column? the filename? is filename-year offset from
   data-year?).
7. **Records source provenance** — URL, retrieval date, method — so
   re-acquisition is never tribal knowledge.
8. **Generates SHA-256 checksums** for every bronze file. These are the freshness
   tripwire: downstream stages refuse to run if bronze changed since this report
   was written.
9. **Writes a Gold Schema Classification table** — for every bronze column, its
   intended role in gold: `fact_key`, `fact_metric`, `fact_categorical`,
   `dimension_attribute`, or `not_in_gold`. This is the blueprint the transform
   implements.

The report is the contract between "understanding the source" and "writing the
code." It is also what the reviewers later use to confirm every source column
was accounted for.

## 6. Stage 2 — `/transform-topic`

**Skill: `transform-topic`. Reads `data-cleaning-standards`. Output:
`transform.py` → gold Parquet + ODCS contract + `_transform_manifest.json` +
`_validation.json`.**

This is the heart of the process. One successful run of `transform.py` produces,
in a single pass: the cleaned gold Parquet, the data contract, a manifest of
everything the transform did, and a passing validation report — and it **exits
non-zero if any validation check fails.** Self-validation is the mechanism that
makes "the transform is correct" a machine-checkable property rather than a
promise.

### The freshness gate (before any code is written)

The skill first runs `scripts/check_bronze_freshness.py`. This **halts** — it
does not warn — if:

- there is no `bronze-data-structure.md` (Stage 1 wasn't run), or
- bronze checksums changed since Stage 1, or
- there are **unanalyzed** bronze files (new files Stage 1 never saw).

Unanalyzed files can never be overridden. New source data *must* go back through
Stage 1 before any transform ingests it.

### Authoring the transform

The skill reads, in order: the bronze structure report, the domain conventions
(`src/etl/{main_topic}/CLAUDE.md`), the universal `data-cleaning-standards`, the
most similar already-completed transform (for consistency), and the shared
utility modules. It then authors a `transform.py` following a fixed pipeline
shape (`main()`):

1. **Read + transform each bronze file** — with read-loss accounting (raw rows
   vs parsed rows), routed by **era detection on column signatures** (not
   hardcoded year ranges). Each per-file function reshapes to the standard gold
   columns and records its categorical recodes on the manifest.
2. **Harmonize + concat** across eras into one DataFrame.
3. **Collision guard, then dedup** — `assert_no_natural_key_collisions()` raises
   if two rows share a natural key but carry *divergent* metrics (a sign of an
   aliasing/aggregation bug); only then does it deduplicate, using an
   **explicitly chosen, commented tie-break rule** (which era wins on overlap
   years, and why).
4. **Geography nulling + known-bad masking** — aggregate rows get NULL geography
   keys; provably-impossible values (an ACT score > 36, a proportion outside
   `[0,1]`) are **NULLed with the row preserved and the masked count recorded**.
   Known-bad data is nulled, never dropped; extreme-but-conceivable values are
   preserved and documented.
5. **Manifest stats + export** — per-year counts, metric summaries, then write
   year-partitioned Parquet.
6. **Emit the contract** — `write_data_dictionary()` projects the in-code column
   declaration into the ODCS contract. The author hand-writes only a few things
   per metric column: its `unit` (`count | proportion | ratio | score | rating |
   currency | percentile`, with optional bounds), which single column is the
   `key_metric` (the headline number most users want), and `metric_component`
   (`numerator`/`denominator`) on the counts that compose a rate/average. The
   emitter **auto-derives** everything else: the row grain, foreign keys,
   range-check SQL, a deterministic `schema_hash`, and example queries.
7. **Validate — always last.** `run_topic_validation()` checks the gold just
   written against the contract just emitted (schema↔contract↔parquet
   conformance, unit/scale rules, grain uniqueness, the topic's own quality SQL,
   foreign-key integrity against the dimensions, canonical vocabulary, geography
   nulling). Any failure → `GoldValidationError` → non-zero exit.

### The manifest — provenance for the review

Alongside gold, the transform writes `_transform_manifest.json`: every
categorical recode map (with unmapped-value detection), per-year row counts,
read-loss events, filtered/masked-row events, and metric summaries. This is not
optional bookkeeping — the next stage's reviewer reads the manifest to verify
**100% of the categorical mappings** and to reconcile row counts. If any
categorical has an unmapped value, `manifest.write()` itself raises.

### Blocking gates in this stage

The transform **halts** (never warns) on: stale/unanalyzed bronze, any unmapped
categorical, any failed validation check, a missing expected bronze column
(silent NULLing — the most common data-loss bug), and natural-key collisions
with divergent metrics. "Data quality > code quality" is the standing rule: you
never weaken a check to make a run pass.

The skill finishes by self-reviewing the new code against a checklist
(`code-review-checklist.md`) for shared-utility usage and code quality — a
separate concern from data accuracy, which the next stage owns.

## 7. Stage 3 — Two independent data reviews

Validation proves the gold is *structurally* consistent with its contract. It
cannot prove the gold is *semantically faithful* to bronze — that a mapping
means the right thing, that a value survived intact, that the era routing and
dedup tie-breaks were the right calls. That judgment layer is two **independent
reviews, run in parallel by two different agents**, so the same blind spot is
unlikely to pass both.

### 3a — `/data-review-claude`

**Skill: `data-review-claude`. Read-only except for its own report. Output:
`data-review-claude.md`.**

A manifest-driven accuracy audit. It explicitly does **not** re-run the
validator's structural checks (it reads `_validation.json` instead). Its job is
the things code can't see:

- **Manifest-driven categorical verification** — for every recode in the
  manifest, is `bronze_value → gold_value` *semantically* correct (not just
  snake_cased)? This is 100% coverage and the highest-value part of the review.
- **A fixed set of risk hypotheses** that have actually shipped as bugs before —
  Asian/PI conflation, column-rename typos (a column ~100% NULL in one era),
  sentinel year-attribution, derived-aggregate errors, dedup tie-break
  inversion, demographic mutual-exclusivity violations, semantically-wrong
  mappings — each ruled in or out with executed evidence.
- **Value-level spot checks** — trace the extreme (max/min) rows and one
  ordinary entity per era from bronze → expected transform → gold. Every verdict
  must quote the bronze line it rests on; "a verdict without a quoted bronze
  line is not acceptable."
- **v1 parity** (project-specific) — where a prior-generation baseline exists,
  compare hashes; a difference isn't a failure but *must be explained*.

Uncertainty is never silently dropped — it is routed to a `NEEDS_JUDGMENT`
section for a human. Confirmed problems go to `## Required Fixes` with five
fixed fields (Severity, Issue, Evidence, Location, Suggested fix).

### 3b — `data-review-codex`

**Skill: `data-review-codex` (run via a different agent, Codex). Output:
`data-review-codex.md`.**

A second, stricter, independent audit run by a *different model/agent* against
the same artifacts — an accounting-style review that insists every source row,
key, metric, and categorical value has a traceable disposition (preserved,
validly reshaped/filtered/excluded, or incorrectly lost/duplicated/fabricated).
It writes to the same `## Required Fixes` schema so the fix loop can dedupe
findings across both reports. The Codex review never blocks the pipeline — the
Claude review is authoritative — but a second pair of eyes with no shared
context catches what one agent rationalizes past.

Running two reviewers with different priors, over the same evidence, is a
deliberate design choice: it is cheap relative to shipping a silent data error.

---

## 8. Stage 4 — `/fix-from-reviews`

**Skill: `fix-from-reviews`. Output: an edited `transform.py`, regenerated gold,
verified fixes.**

This skill consolidates the `## Required Fixes` from both review reports, applies
the mechanical ones to `transform.py`, **re-runs the transform** (which
re-emits everything and re-validates), and then **verifies each fix actually
landed in the regenerated gold** — verification is part of the skill, not a
separate step. Because fixes go into the transform and the transform is
re-run, the fix can never be a one-off patch to gold that the next rebuild
silently reverts.

`NEEDS_JUDGMENT` items are *not* auto-applied — in interactive mode the user is
asked; in batch mode (`--no-prompt`) they are deferred to a report and the
conservative choice (preserve bronze granularity) is taken. It is a single pass:
the reviews are not re-run afterward, and any remaining items surface in the
summary for a human.

---

## 9. Stage 5 — `/approve-topic` (user only)

**Skill: `approve-topic`. Only the user runs it — never an agent unprompted.**

Approval is the human sign-off. After the user has personally inspected the gold
Parquet and both review reports, `/approve-topic` flips `approved: true` in
`topic-status.yaml` and captures **hash baselines** — the topic's
`approved_gold_sha256` plus the dimension hashes — so any later drift (a
gold edit, a dimension rebuild) is detectable. The script refuses to approve
unless a fresh, passing `_validation.json` exists; you cannot approve a topic
whose transform doesn't currently pass.

Flipping that one flag is what makes the topic live: the API, MCP server,
dashboard, and chatbot all read `topic-status.yaml` + the contracts at startup,
so an approved topic is picked up automatically on the next restart with **no
per-topic code** anywhere downstream.

## 10. Orchestrators — running the pipeline at scale

The single-topic skills compose:

- **`/full-pipeline <main> <sub> <topic>`** runs the whole chain for one topic:
  freshness gate → transform (in its own agent) → independent validation
  re-check → **parallel** Claude + Codex reviews → conditional fix loop. It
  never approves — that stays with the user. Each heavy stage runs in a fresh
  agent context so a reviewer isn't grading its own homework.
- **`/batch-pipeline <main> <sub>`** fans `/full-pipeline` across every eligible
  topic in a sub-topic, in parallel non-prompting batches, deferring all
  judgment calls to one aggregate report instead of asking mid-run.
- **`/next-topic`** walks `topic-status.yaml` alphabetically and runs
  `/full-pipeline` on the first topic that has a bronze structure report but no
  transform yet — so "run the next one" needs no arguments.
- **`/pipeline-status`** reports the stage matrix across all topics, plus drift
  and referential-integrity status. This is the "where am I" dashboard.

## 11. The safety net, summarized

The reason this process can run largely autonomously is that the failure modes
are turned into **hard gates**, not warnings:

| Guard | Enforced by | Effect |
|-------|-------------|--------|
| Bronze changed / unanalyzed | `check_bronze_freshness.py` | Transform refuses to run |
| Unmapped categorical value | `manifest.write()` | Transform raises |
| Missing expected bronze column | transform's own column check | Halt (prevents silent NULLing) |
| Natural-key collision w/ divergent metrics | `assert_no_natural_key_collisions()` | Transform raises |
| Structural drift from contract | `run_topic_validation()` | Non-zero exit |
| Semantic mapping error | `/data-review-claude` + `data-review-codex` | Required Fix |
| Post-approval gold/dimension drift | `check_approved_topics.py` (hash baselines) | Detected + reported |

Two rules sit above all the gates: **never edit gold directly** (change the
transform and re-run), and **never weaken a check to make a run pass** (data
quality outranks code quality, always).

## 12. Implementing this yourself with Claude Code

You do not need this project's exact schema to adopt the pattern. The
transferable structure is:

1. **Two immutable-source / regenerable-output tiers.** Keep raw source files
   untouched; make the cleaned output a pure function of a transform script.
   Never hand-edit the output.

2. **A self-validating transform.** The single most important idea. Have the
   transform emit a machine-readable schema/contract *and* validate its own
   output against that contract as its final act, exiting non-zero on any
   failure. This turns "is the data clean?" into a build step.

3. **A written source-profiling artifact before any code.** Encode the profiling
   as a skill (`/bronze-data-structure` here) so it is done the same way every
   time and produces the blueprint the transform implements.

4. **Skills for each stage, and orchestrators that compose them.** Author,
   review, fix, and approve as separate skills run by separate agents. Write the
   non-negotiable gates *into* the skills so they can't be skipped.

5. **A judgment-layer review distinct from structural validation.** Code can
   check shapes and ranges; it can't check meaning. Run one (ideally two,
   independent) accuracy audits that trace values back to source and quote the
   evidence.

6. **A rules reference skill.** Put the universal conventions (naming, types,
   scaling, the domain traps) in one skill that the authoring and reviewing
   skills both read. This is where your domain expertise lives; it is the file
   you will edit most as you learn your data.

Concretely, to start: create `.claude/skills/<name>/SKILL.md` files with YAML
frontmatter (`name`, `description`, `allowed-tools`) and a numbered procedure
including the exact commands and the stop-conditions. Point the authoring skill
at your standards skill. Wire them into an orchestrator once the single-topic
flow is solid. The skills in `.claude/skills/` of this repo are working
reference implementations you can read and adapt directly.
